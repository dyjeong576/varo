# VARO PRD

## 1. Executive Summary

VARO(Verified Analysis, Reasoned Opinion)는 사용자가 뉴스, 기사, 온라인 게시물, 인터뷰 발언처럼 사실성 검토가 필요한 `check`을 입력하면 관련 `evidence`를 수집하고, 그 근거만을 기반으로 `interpretation`을 구조화해 보여주는 서비스다.

브랜드 표기는 기본적으로 `VARO`를 사용하고, 사용자-facing 포지셔닝 문구는 `Verified Analysis, Reasoned Opinion`으로 유지한다.

MVP의 목적은 "진실을 판정"하는 것이 아니라, 사용자가 직접 판단할 수 있도록 근거와 맥락을 정리해 제공하는 것이다. 따라서 결과는 항상 수집된 출처 기준의 해석이어야 하며, 불확실성이 남아 있으면 이를 숨기지 않는다.

이번 MVP는 한국어 뉴스 소비자를 주요 대상으로 하며, 검토 범위는 **한국 정치·경제** 도메인의 사실성 check으로 고정한다. 한국 정치·경제 뉴스성 check은 네이버 뉴스 검색 API를 기본 검색 경로로 사용하고, 네이버 후보가 충분하지 않을 때만 Tavily Search를 보조 검색 provider로 사용한다. 현재 preview 생성 경로에서는 원문 본문 추출보다 검색 결과의 제목과 스니펫, source metadata, evidence signal을 우선 사용한다. 한국 정치·경제 범위를 벗어나는 check은 지원 범위 밖으로 안내한다. 핵심 경험은 결과 페이지이며, 결론보다 근거를 먼저 이해할 수 있게 설계한다.

## 2. Product Vision

VARO의 장기 비전은 사용자가 어떤 주장에 대해서도 "왜 이런 해석이 나왔는지"를 출처와 함께 추적할 수 있는 근거 중심 검토 인터페이스를 제공하는 것이다.

제품은 아래 원칙을 유지해야 한다.

- 절대적 진실 판정기처럼 행동하지 않는다.
- `check`, `evidence`, `interpretation`, `uncertainty`를 분리해 보여준다.
- 사용자가 원문 출처까지 직접 확인할 수 있게 한다.
- 상충하는 근거를 숨기지 않는다.

## 3. Problem

뉴스와 온라인 정보 소비 환경에서는 같은 주장에 대해 여러 기사, 해설, 재인용 게시물이 빠르게 확산된다. 일반 사용자는 아래 문제를 겪는다.

- 어떤 문장이 검토 대상 주장인지 분리하기 어렵다.
- 여러 기사를 읽어도 원출처와 재인용을 구분하기 어렵다.
- 기사 수가 많아 보여도 실제로는 같은 정보가 반복될 수 있다.
- 공식 발표, 언론 보도, 해설 기사, 소셜 반응이 한 화면에서 비교되지 않는다.
- 사용자의 발화에 포함된 단어가 실제 검증에 필요한 검색 질의와 다를 수 있다.
- AI 요약 서비스는 결론은 주지만 근거 구조를 충분히 보여주지 않는 경우가 많다.

## 4. Target Users

### Primary User

정치·경제 뉴스와 온라인 이슈를 자주 접하는 한국어 뉴스 소비자 (한국 정치·경제 뉴스 전용)

특성:

- 헤드라인이나 게시물만으로 판단하기 불안하다.
- 여러 출처를 직접 찾고 비교하는 데 시간이 많이 든다.
- 결론보다 근거와 출처를 확인하고 싶다.

### Secondary User

초기 MVP 기준 보조 타깃은 아래 사용자다.

- 정치·경제 이슈를 빠르게 이해하려는 직장인
- 뉴스 리터러시를 높이고 싶은 학생
- 가족이나 지인과 공유된 주장에 대해 근거를 확인하려는 사용자

## 5. User Pain Points

- "이 주장이 실제로 어떤 근거 위에서 돌고 있는지" 파악하기 어렵다.
- 기사 제목만 비슷하고 내용 차이가 큰 경우를 구분하기 어렵다.
- 속보와 후속 정정, 공식 해명과 언론 해석을 한 번에 보기 어렵다.
- AI가 단정적으로 말하면 왜 그런 결론이 나왔는지 확인하기 어렵다.
- 신뢰하기 어려운 출처가 섞여 있어도 사용자가 이를 바로 감지하기 어렵다.

## 6. Value Proposition

VARO는 사용자가 하나의 주장에 대해 다음을 빠르게 확인하도록 돕는다.

- 어떤 `check`이 검토 대상인지
- 해당 check을 검증하기 위해 어떤 검색 관점이 필요한지
- 어떤 `evidence`가 수집되었는지
- 수집된 출처를 기준으로 어떤 `interpretation`이 가능한지
- 무엇이 아직 `uncertainty`로 남아 있는지

핵심 가치는 "결론 자동화"가 아니라 "근거 구조화"다.

## 7. MVP Scope

MVP에서 포함하는 범위는 아래와 같다.

- 사용자가 검토하고 싶은 `check` 입력
- check의 정치·경제 도메인 적합성 자동 판정
- check의 뉴스성 및 검색 provider route 확인
- check 이해 기반 search plan 생성
- 관련 출처 수집
- 출처 메타데이터 표시
- 검색 결과 스니펫과 evidence signal 기반 근거 표시
- 수집된 출처만 기반으로 한 rule-based preview 요약 및 verdict 표시
- 출처 간 일치, 충돌, 불확실성 표시
- 결과 페이지 중심 UX

MVP 기준 검토 가능 check:

- 한국 정치·경제 뉴스성 check
- 한국 정치인 발언, 정당/정부 입장, 정책, 공약, 법안, 예산, 세금, 물가, 금리, 부동산, 기업 공식 발표, 공시, 경제 지표처럼 사실성 검토가 가능한 check

MVP 기준 현재 출처 범위:

- 한국 뉴스 기사
- 뉴스 검색 결과에서 확인되는 정부, 기관, 기업, 단체의 발표/보도
- 주요 소셜 원문과 직접 공식 문서 수집은 후속 확장 범위다.

## 8. Out of Scope

이번 MVP에서 제외하는 범위는 아래와 같다.

- 완전한 다국어 지원
- 뉴스성 또는 사실성 검토 대상이 아닌 일반 의견/상담/창작 요청
- 지원 범위 밖 뉴스 check
- 의료, 연예, 스포츠 등 정치·경제 밖의 도메인
- 가치판단, 정쟁성 의견, 미래 예측, 투자 매수/매도 추천
- 검색 provider로 근거 수집이 불가능한 check에 대한 verdict 생성
- SNS 전체 네트워크 추적
- 사람 또는 조직별 신뢰도 점수 체계
- 법률 수준의 사실 인증
- 개인화된 정치 성향 분석
- 사용자별 추천 피드
- 복잡한 편집자 워크플로우
- 상세 API 스펙, 상세 DB 스키마, 운영 대시보드

## 9. User Stories

- 사용자는 검증하고 싶은 주장을 입력하고 관련 근거를 한 번에 보고 싶다.
- 사용자는 결과 페이지에서 출처 간 일치와 충돌을 빠르게 파악하고 싶다.
- 사용자는 각 출처의 발행 시각, 출처 유형, 링크, 핵심 스니펫을 확인하고 싶다.
- 사용자는 AI 결론만 보는 것이 아니라 어떤 근거를 바탕으로 그런 해석이 나왔는지 알고 싶다.
- 사용자는 근거가 부족하거나 상충될 때 서비스가 이를 숨기지 않기를 원한다.

## 10. Core Features

### 10.1 Check Input

- 사용자가 사실성 검토가 필요한 문장을 입력한다.
- 입력 단위는 `check`이다.

### 10.2 Source Collection

- 시스템이 관련 `source`를 수집한다.
- 현재 수집 범위는 Naver News Search와 필요 시 Tavily Search fallback으로 확인되는 뉴스 source다.
- 시스템은 사용자 발화에서 추출한 단어를 그대로 검색하지 않고, check 검증 목적에 맞는 search plan을 생성한다.
- 기본 검색 목적은 `check_specific`, `current_state`, `primary_source`, `contradiction_or_update`로 구분한다.
- 시스템은 뉴스 검색에서 Naver News Search를 먼저 사용하고, 후보가 부족할 때만 Tavily Search fallback을 사용한다.

### 10.3 Evidence Preparation

- 각 출처의 제목, 검색 결과 스니펫, relevance reason, evidence signal을 근거 맥락으로 표시한다.
- 본문 추출 기반 `evidence snippet`은 후속 확장 대상으로 두되, 현재 preview에서는 source-level signal을 우선 사용한다.
- 사용자에게 어떤 source와 어떤 signal이 결과에 쓰였는지 추적 가능하게 제공한다.

### 10.4 Structured Answer Result

- 수집된 출처와 evidence signal을 기반으로 preview result를 생성하고, 핵심 해석과 불확실성 summary는 OpenAI structured output으로 만든다.
- 결과에는 verdict, 핵심 해석, 불확실성이 포함된다.

### 10.5 Source Comparison

- 출처 간 일치, 충돌, 부족한 부분을 시각적으로 구분한다.
- 기사 수가 아니라 근거 성격과 출처 유형 차이를 함께 고려한다.
- 출처 간 일치도는 사실 확률이 아니며, 최신성, 상충, 업데이트 신호와 분리해 해석한다.

### 10.6 Today Headlines

- 시스템은 매일 새벽 1시(KST)에 주요 매체 RSS의 제목, 링크, 요약, 발행시각을 수집해 저장한다.
- 시스템은 정치 RSS와 경제 RSS를 수집해 저장한다.
- 사용자는 `/headlines`에서 정치/경제 메뉴와 날짜 검색으로 수집된 매체별 헤드라인을 비교할 수 있어야 한다.
- 날짜 검색은 헤드라인 수집 시작일인 2026-04-30 이후 날짜만 선택할 수 있어야 한다.
- `/headlines`의 기사 카드에는 매체별 RSS 발행시각을 표시하지 않는다.
- `/headlines`는 선택한 날짜와 카테고리에서 수집된 기사를 일부로 제한하지 않고 모두 노출한다.
- 분석 API는 저장된 RSS 제목/요약/매체명만 기반으로 사건별 묶음과 매체별 표현 차이를 제공한다.
- 오늘의 헤드라인 분석 summary는 짧은 개요 문단과 핵심 bullet로 주요 흐름, 반복 이슈, 표현 차이, 불확실성을 설명한다.
- 오늘의 헤드라인 분석은 사실 판정이나 매체 신뢰도 점수가 아니라 수집된 헤드라인 기준의 표현 비교로 제한한다.

## 11. Functional Requirements

### 11.1 Input and Answer Flow

- 사용자는 하나의 `check`을 입력할 수 있어야 한다.
- 시스템은 입력 후 검토 진행 상태를 보여줄 수 있어야 한다.
- 시스템은 정치·경제 뉴스성 check인지 자동 판정해야 한다.
- 시스템은 check의 `search_route`를 판정하고, `unsupported`이면 verdict를 만들지 않고 지원 범위 밖 상태로 기록해야 한다.
- 현재 구현에서 공개 API 기준 authoritative field는 `search_route`이며, 판정 이유는 `search_route_reason`으로 설명해야 한다.
- 시스템은 query refinement에서 출처 기반 사실성 검토 대상 여부를 `isFactCheckQuestion` boolean으로 분류해야 한다.
- 시스템은 `search_route`와 별도로 검증 목적별 `search_plan`을 생성해야 한다.
- 시스템은 Naver 후보가 충분하면 Tavily를 호출하지 않아야 한다.
- 사용자는 검토가 끝난 뒤 결과 페이지로 이동할 수 있어야 한다.

### 11.2 Source Display

- 각 출처 카드는 최소 아래 정보를 포함해야 한다.
  - 매체명 또는 출처명
  - 발행 시각
  - URL
  - 핵심 스니펫
  - 출처 유형
- 한국 뉴스사 성향 배지는 출처 맥락을 돕는 보조 메타데이터로 표시하며, verdict나 출처 신뢰도 점수로 사용하지 않는다.
- 사용자는 각 출처 원문 링크로 이동할 수 있어야 한다.

### 11.3 Verdict and Interpretation

- 시스템은 아래 4단계 verdict 중 하나를 표시해야 한다.
  - `Likely True`
  - `Mixed Evidence`
  - `Unclear`
  - `Likely False`
- verdict는 항상 수집된 출처 기준 해석으로 표현해야 한다.
- 결과에는 짧은 해석 요약과 함께 남아 있는 불확실성을 포함해야 한다.

### 11.4 Evidence Comparison

- 시스템은 근거 간 일치와 충돌을 사용자가 스캔 가능하게 표시해야 한다.
- 시스템은 출처 합의성, 최신성, 업데이트/정정 신호를 별도 개념으로 다뤄야 한다.
- 출처 부족 시 `Unclear`를 우선 고려해야 한다.
- 공식 발표와 언론 기사 해석이 충돌하는 경우 충돌 상태를 숨기지 않아야 한다.

### 11.5 Traceability

- 각 결과는 어떤 `check`에 대한 것인지 연결되어야 한다.
- 각 해석은 어떤 `source`와 evidence signal을 기반으로 했는지 추적 가능해야 한다. snippet row가 있는 경우에는 snippet까지 연결해야 한다.

### 11.6 Today Headlines Flow

- 시스템은 RSS 수집 실행 상태, 수집 기사 수, 저장 기사 수, 오류 메시지를 기록해야 한다.
- RSS 중복 저장 기준은 `publisher_key + normalized_url`로 둔다.
- `GET /api/v1/headlines/today`는 날짜, 마지막 수집 상태, 전체 기사 수, 매체별 기사 목록을 반환해야 하며 `date`, `category=politics|economy` 필터를 지원해야 한다.
- `GET /api/v1/headlines/live`는 DB 저장 없이 RSS를 즉시 조회해 매체별 기사 목록을 반환해야 하며 `category=politics|economy` 필터를 지원해야 한다.
- `GET /api/v1/headlines/today/analysis`는 분석 상태와 사건별 cluster를 반환해야 하며 `category=politics|economy` 필터를 지원해야 한다.
- 헤드라인 분석 결과는 정치/경제 분석이 섞이지 않도록 `date + category` 단위로 저장하고 조회해야 한다.
- 수동 수집 API는 내부 secret header로 보호해야 하며 `category=politics|economy`로 정치 또는 경제 RSS만 수집할 수 있어야 한다.

## 12. Non-Functional Requirements

- 결과는 차분하고 과장 없는 톤으로 표현해야 한다.
- 정보 구조는 사용자가 짧은 시간 안에 근거를 스캔할 수 있어야 한다.
- 결과 페이지는 모바일과 데스크톱에서 모두 읽기 쉬워야 한다.
- 링크, 발행 시각, 출처 유형 등 핵심 메타데이터는 시각적으로 누락되면 안 된다.
- 제품 전반에서 "사실 확정"이 아니라 "근거 기반 검토"라는 메시지가 유지되어야 한다.

## 13. UX Principles

- 근거가 결론보다 먼저 보여야 한다.
- 결론은 과장 없이 요약형 문장으로 제시한다.
- 상충하는 근거는 별도로 표시하고 숨기지 않는다.
- 사용자가 원문 출처를 직접 확인할 수 있어야 한다.
- 스니펫, 출처명, 발행 시각이 빠르게 스캔 가능해야 한다.

## 14. Information Architecture

핵심 개념 구조는 아래와 같다.

| 단위               | 설명                                                    |
| ------------------ | ------------------------------------------------------- |
| `check`            | 사용자가 검토하고 싶은 문장 또는 주장                   |
| `source`           | 수집된 기사 또는 뉴스 검색 결과에서 확인된 출처         |
| `evidence signal`  | 각 source가 check에 대해 수행하는 지지/충돌/맥락 역할   |
| `answer result`    | verdict, interpretation, uncertainty를 포함한 결과 단위 |
| `headline article` | RSS에서 수집한 매체별 기사 제목/요약/링크 단위          |
| `headline cluster` | 여러 매체 헤드라인을 사건별로 묶은 표현 비교 단위       |

페이지 구조는 아래와 같다.

- Check 입력 페이지
- 검토 진행 상태 페이지
- 결과 페이지
- 인기 주제 페이지
- 오늘의 헤드라인 페이지

## 15. Screen-by-Screen Spec

### 15.1 Check 입력 화면

목적:

- 사용자가 검토하고 싶은 주장을 명확히 입력한다.

포함 요소:

- check 입력 필드
- 검토 시작 CTA
- 서비스 설명 문구

핵심 UX 포인트:

- 사용자가 검토 대상 문장을 명확히 인식할 수 있어야 한다.
- 서비스가 절대적 사실 판정기가 아니라는 점을 짧게 안내한다.

### 15.2 검토 진행 / 수집 상태 화면

목적:

- 시스템이 출처를 수집하고 있다는 점을 투명하게 보여준다.

포함 요소:

- 현재 검토 중인 check 요약
- 수집 및 분석 진행 상태
- 결과 페이지로 이어지는 전환 상태

핵심 UX 포인트:

- 사용자는 "검토 중"이라는 상태를 이해할 수 있어야 한다.
- 실제 출처 기반 결과가 생성된다는 기대를 형성해야 한다.

### 15.3 결과 화면

목적:

- 사용자가 근거를 먼저 이해하고, 그 다음 해석과 verdict를 보게 한다.

포함 요소:

- check 요약
- verdict
- interpretation 요약
- uncertainty 요약
- evidence agreement / conflict / uncertainty 표시
- source list
- 원문 링크 이동

핵심 UX 포인트:

- 근거가 결론보다 먼저 보이는 정보 구조여야 한다.
- source card는 스캔하기 쉬워야 한다.
- 사용자는 충돌하는 근거와 부족한 근거를 바로 식별할 수 있어야 한다.

### 15.4 인기 주제 화면

목적:

- 최근 24시간 동안 사용자 관심이 반복적으로 모인 주제를 빠르게 훑고 기존 answer preview로 재진입하게 한다.

포함 요소:

- topic 순위
- topic 텍스트
- 최근 24시간 `submitted + meaningful reopen` 합산 점수
- 요청 수와 재열람 수 분리 표시
- 대표 answer preview 재진입 링크

핵심 UX 포인트:

- popularity는 단순 조회수가 아니라 새 검토 요청과 의미 있는 재진입의 합산임을 명시해야 한다.
- 인기 항목 선택 시 새 검토를 만들지 않고 기존 answer preview로 바로 들어가야 한다.
- 인기 항목은 근거를 다시 확인하는 진입점이어야 하며, 결론 소비 화면처럼 과장되게 보이면 안 된다.

## 16. Verdict / Evidence Model

### 16.1 Verdict States

MVP의 verdict는 아래 네 단계로 고정한다.

- `Likely True`
- `Mixed Evidence`
- `Unclear`
- `Likely False`

운영 원칙:

- 이분법적 진실/거짓 표현을 기본값으로 사용하지 않는다.
- 출처가 부족하거나 모호하면 `Unclear`를 우선 고려한다.
- 상충하는 근거가 충분하면 `Mixed Evidence`를 사용한다.
- 동일 오보 재인용 가능성이 높으면 기사 수만으로 `Likely True`를 주지 않는다.

### 16.2 Evidence Model

각 결과는 아래를 분리해서 다룬다.

- `Check`: 사용자가 입력한 검토 대상 문장
- `Evidence`: 수집된 출처와 스니펫, 메타데이터
- `Interpretation`: 수집된 출처만 기반으로 한 요약과 해석
- `Uncertainty`: 남아 있는 모호함, 충돌, 부족한 근거

### 16.3 Source Type Concept

MVP에서 최소한 아래 출처 유형을 구분할 필요가 있다.

- 언론 기사
- 공식 발표
- 소셜 원문 (후속 확장)
- 해설/분석
- 재인용 가능

이 구분은 정교한 신뢰도 점수 대신 "출처 성격 차이"를 사용자에게 보여주기 위한 최소 단위다.

## 17. Trust & Safety

- VARO는 절대적 진실을 선언하지 않는다.
- 결과는 수집된 출처 기준의 해석으로 표현한다.
- 출처가 적거나 상충되면 불확실성을 명시한다.
- 공식 발표, 기사, 해설, 소셜 원문을 같은 성격으로 취급하지 않는다. 현재 preview 경로는 주로 뉴스 source를 다룬다.
- 기사 수가 많다고 사실성이 높다고 단정하지 않는다.
- 동일 오보 재인용 가능성을 리스크로 다룬다.
- 속보와 정정 기사 차이를 고려해야 한다.
- 사용자가 직접 원문을 열어볼 수 있는 링크를 제공한다.

## 18. Metrics

초기 MVP에서 추적할 핵심 지표는 아래와 같다.

- check 제출 수
- 결과 페이지 도달률
- 출처 카드 클릭률
- 원문 링크 클릭률
- 결과 페이지 체류 시간
- verdict별 분포
- `Unclear` 비중

이 지표의 목적은 "많이 맞췄는가"보다 "사용자가 근거를 실제로 탐색했는가"를 확인하는 데 있다.

## 19. Risks / Open Questions

### Risks

- 같은 오보가 여러 기사에서 반복 재인용될 수 있다.
- 공식 발표와 언론 기사 해석이 충돌할 수 있다.
- 속보 이후 정정이 나와도 초기 기사만 많이 노출될 수 있다.
- 소셜 원문은 확산 속도가 빠르지만 진위와 맥락이 불안정하므로 후속 확장 시 별도 검증 정책이 필요하다.
- 사용자가 verdict만 보고 근거를 충분히 읽지 않을 수 있다.

### Open Questions

- MVP에서 포함할 "주요 소셜 일부"의 구체 플랫폼 범위는 무엇인가.
- 네이버 뉴스 검색 결과와 공식 발표/원문 링크를 어떤 우선순위로 함께 보여줄 것인가.
- 결과 생성 속도와 출처 수집 깊이 사이의 우선순위를 어떻게 둘 것인가.
- 출처 유형 표기를 사용자에게 얼마나 단순하게 보여줄 것인가.

## 20. Recommended MVP Decisions

- 초기 타깃은 **한국 정치·경제 뉴스**를 소비하는 한국어 사용자로 둔다.
- 한국 정치·경제 뉴스성 check은 네이버 뉴스 검색 API를 기본 검색 provider로 사용한다.
- Tavily Search는 Naver 후보가 부족할 때만 한국 뉴스 보조 검색 provider로 사용한다.
- Tavily Extract는 후속 본문 추출 확장용으로 유지하되, 현재 preview 생성 경로에서는 사용하지 않는다.
- 핵심 경험은 결과 페이지 중심으로 설계한다.
- 입력은 단일 check 중심으로 단순하게 시작한다.
- verdict는 4단계 모델을 유지한다.
- source card에는 출처명, 발행 시각, URL, 핵심 스니펫, 출처 유형을 기본 포함한다.
- source card의 뉴스사 성향 배지는 출처 맥락 표시용이며, 사실성 판단 점수가 아니다.
- 근거 부족 시 `Unclear`를 우선 선택한다.
- 기사 수를 신뢰도의 대리값으로 사용하지 않는다.
- 소셜 출처 직접 수집은 후속 단계에서 결정한다.

## 21. Roadmap

### Phase 1: MVP Launch

- check 입력
- 출처 수집
- source card 표시
- verdict 및 interpretation 표시
- agreement / conflict / uncertainty 표시

### Phase 2: Better Source Understanding

- 출처 유형 분류 정교화
- 정정 기사 및 후속 보도 연결 강화
- 공식 원문과 해설 기사 연결 개선

### Phase 3: Expanded Verification Experience

- 다국어 확장 검토
- 더 넓은 소셜 범위 검토
- 사용자 저장, 공유, 히스토리 기능 검토
