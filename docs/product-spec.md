# VARO Product Spec

## 1. 문서 개요

### 1.1 목적

이 문서는 VARO 서비스의 화면 구조, 사용자 흐름, 주요 페이지 요구사항, UI/UX 방향을 정의한다.  
VARO는 사용자가 한국 관련 정치·경제 뉴스, 주장, 이슈에 대해 질문하면 네이버 뉴스 검색 API를 우선 사용해 관련 기사를 수집하고, 네이버 후보가 충분하지 않을 때만 Tavily Search 보조검색을 사용해 사용자가 객관적으로 판단할 수 있도록 돕는 모바일 중심 반응형 웹 서비스다. 현재 preview 생성 경로는 원문 본문 추출보다 검색 결과 스니펫과 source metadata, evidence signal을 우선 사용한다. 해외뉴스 요청은 현재 한국뉴스만 분석한다는 안내와 함께 지원 범위 밖으로 처리한다.
브랜드는 `VARO`를 기본 표기로 사용하고, 사용자-facing 포지셔닝 문구는 `Verified Analysis, Reasoned Opinion`으로 유지한다.

### 1.2 범위

본 문서는 다음 범위를 포함한다.

- Information Architecture
- Navigation Structure
- User Flow
- Page Specification
- UI/UX 방향
- 기본 정책

### 1.3 제품 방향

- 모바일 웹 우선 설계
- 반응형 웹 지원
- 신뢰감 있는 정보 탐색 경험 제공
- 복잡한 기능보다 빠른 질문, 분석, 확인 흐름에 집중
- 초기 버전에서는 다크모드 미지원

## 2. 기본 정책

### 2.1 플랫폼

- 서비스 형태: 모바일 웹
- 지원 방식: 반응형 웹
- 우선 기준: 모바일 퍼스트
- 데스크톱에서도 사용 가능해야 하나, 핵심 UX는 모바일 기준으로 설계한다.

### 2.2 인증 정책

- 로그인은 필수다.
- 홈 진입 시 비로그인 상태라면 로그인 페이지로 리다이렉트한다.
- 로그인 방식은 **구글 로그인만 지원**한다.
- 초기 버전에서는 이메일 로그인, 자체 회원가입, SNS 추가 로그인은 지원하지 않는다.

### 2.3 테마 정책

- 대표 컬러는 **신뢰감을 주는 중간 톤의 블루**를 사용한다.
- 너무 어둡거나 너무 밝은 블루는 지양한다.
- 다크모드는 지원하지 않는다.
- 전체 UI는 명확하고 밝은 기본 테마를 유지한다.

### 2.4 커뮤니티 정책

- 커뮤니티에서는 익명 사용을 허용하지 않는다.
- 모든 사용자는 **실명, 성별, 나이대**를 공개한 상태로 활동해야 한다.
- 커뮤니티의 신뢰성과 책임성을 강화하는 방향으로 운영한다.

### 2.5 사용자 정보 정책

- 사용자 정보 화면에서 다음 정보를 확인할 수 있다.
  - 실명
  - 성별
  - 나이
  - 사는 나라
  - 사는 도시
- 이 중 **사는 나라, 사는 도시만 수정 가능**하다.
- 실명, 나이, 성별은 수정할 수 없다.

- Onboarding Profile
- Home

- Popular
- Community

- User Settings / User Info

## 4. Navigation Structure

## 4.1 하단 네비게이션

하단 네비게이션에는 다음 3개만 노출한다.

- Home
- 인기
- 커뮤니티

## 4.2 상단 버튼 구조

상단에는 다음 버튼을 둔다.

- 좌측: 햄버거 메뉴 버튼
- 우측: 알림 버튼

## 4.3 햄버거 메뉴

햄버거 메뉴를 열면 다음 항목을 확인할 수 있다.

- 히스토리
- 설정

설정 화면 내부에는 사용자 정보 확인 버튼이 있다.

현재 구현 기준으로 설정 화면에는 아래 메뉴가 함께 노출된다.

- 내 정보 관리: 동작
- 개인정보 보호: MVP 처리방침 요약 페이지로 동작
- 알림 설정: 서버 저장 기반 설정 페이지로 동작
- 고객 센터: FAQ와 임시 문의 안내 페이지로 동작
- 로그아웃: 세션 종료와 로컬 상태 정리까지 동작

## 4.4 알림 버튼

알림 버튼은 다음 용도로 사용한다.

- 커뮤니티 관련 알림
- 사용자가 요청한 분석 완료 알림

## 5. User Flows

## 5.1 로그인 진입 흐름

1. 사용자가 서비스에 접속한다.
2. 시스템은 로그인 여부를 확인한다.
3. 비로그인 상태라면 로그인 페이지로 이동시킨다.
4. 사용자는 구글 로그인 버튼을 눌러 인증한다.
5. 로그인 성공 시 Home으로 이동한다.

## 5.2 질문 분석 흐름

1. 사용자가 Home 화면 중앙 채팅박스에 질문을 입력한다.
2. 사용자가 질문 전송 액션을 수행한다.
3. 프론트는 전역 review task를 생성하고 Analysis Loading 화면으로 이동한다.
4. active review task가 이미 있으면 Home 대신 기존 Loading 화면으로 복귀시킨다.
5. 시스템은 query refinement, provider route 판정, Naver 우선 source search, 필요 시 Tavily fallback, relevance/evidence signal 통합 분류 기준의 review preview를 생성한다.
6. preview가 준비되면 사용자는 결과 보기 액션을 통해 Analysis Result 화면으로 진입한다.
7. 사용자는 검색 결과 스니펫, source 목록, query context를 먼저 확인한다.

## 5.3 분석 완료 알림 흐름

1. 사용자가 질문을 입력하고 분석을 요청한다.
2. 분석이 일정 시간 이상 소요된다.
3. review preview가 terminal non-failed 상태로 준비되면 서버가 분석 완료 알림을 생성한다.
4. 사용자는 알림 버튼을 통해 완료 여부를 확인할 수 있다.
5. 사용자는 알림을 눌러 결과 화면으로 이동할 수 있다.

## 5.4 인기 화면 탐색 흐름

1. 사용자가 하단 네비게이션에서 인기 탭을 선택한다.
2. 시스템은 최근 24시간 `submitted + meaningful reopen` 점수가 높은 topic을 순위 형태로 보여준다.
3. 사용자는 인기 항목을 눌러 해당 topic의 대표 review preview로 이동할 수 있다.

## 5.5 커뮤니티 이용 흐름

1. 사용자가 하단 네비게이션에서 커뮤니티 탭을 선택한다.
2. 게시글 목록을 확인한다.
3. 게시글 작성자 정보로 실명, 성별, 나이대가 노출된다.
4. 사용자는 게시글 상세 진입, 댓글 작성, 반응 확인 등을 수행한다.
5. 관련 활동이 발생하면 알림으로 전달될 수 있다.

## 5.6 히스토리 및 사용자 정보 확인 흐름

1. 사용자가 상단 좌측 햄버거 메뉴를 연다.
2. 히스토리 또는 설정을 선택한다.
3. 설정 화면에서 사용자 정보 버튼을 누른다.
4. 사용자 정보 화면에서 프로필 정보를 확인한다.
5. 사는 나라, 사는 도시만 수정 가능하다.

## 5.7 온보딩 프로필 입력 흐름

1. 사용자가 구글 로그인에 성공한다.
2. 시스템은 프로필 완성 여부를 확인한다.
3. 필수 프로필이 비어 있으면 `/onboarding/profile`로 이동한다.
4. 사용자는 실명, 성별, 나이대, 국가, 도시를 입력한다.
5. 저장이 완료되면 Home으로 이동한다.

## 6. Page Specifications

## 6.1 Login Page

### 목적

비로그인 사용자를 인증시키고 서비스 진입을 제어한다.

### 주요 요소

- 서비스 로고
- 서비스 한 줄 소개
- 구글 로그인 버튼

### 필수 동작

- 로그인하지 않은 사용자는 Home에 직접 접근할 수 없다.
- 로그인 페이지에서 구글 로그인만 제공한다.
- 로그인 성공 시 Home으로 이동한다.

### 상태

- 기본 상태
- 로그인 진행 중
- 로그인 실패
- 로그인 성공

### UX 메모

- 첫 진입에서 복잡한 설명보다 서비스 신뢰감과 간결함을 우선한다.
- CTA는 구글 로그인 버튼 하나만 강조한다.

---

## 6.1.1 Onboarding Profile Page

### 목적

첫 로그인 사용자가 서비스 이용 전 필수 프로필을 입력하도록 한다.

### 주요 요소

- 실명 입력
- 성별 선택
- 나이대 선택
- 활동 국가 입력
- 활동 도시 입력
- 저장 CTA

### 필수 동작

- 인증된 사용자만 접근 가능해야 한다.
- 프로필이 이미 완성된 사용자는 Home으로 이동한다.
- 저장 완료 후 Home으로 이동한다.

### UX 메모

- 첫 로그인에서 왜 이 정보가 필요한지 간결하게 설명해야 한다.
- 커뮤니티와 분석 이력이 하나의 계정 체계로 연결된다는 점을 안내한다.

## 6.2 Home Page

### 목적

사용자가 정치·경제 claim을 입력하고 분석을 시작하는 핵심 진입 화면이다.

### 주요 요소

- 상단 바
  - 좌측 햄버거 메뉴 버튼
  - 우측 알림 버튼
- 중앙 채팅박스
- 질문 입력 영역
- 질문 전송 액션

### 핵심 구조

- 화면 정중앙에는 채팅박스만 강조한다.
- 사용자가 자연스럽게 질문을 입력하도록 단순한 구조를 유지한다.
- 초기 화면에는 불필요한 정보나 과한 카드 노출을 지양한다.

### 필수 동작

- 질문 입력 가능
- 질문 전송 시 분석 요청 생성
- 정치·경제 밖 주제는 지원 범위 밖으로 안내
- 요청 후 Analysis Loading 화면으로 이동

### 상태

- 입력 전
- 입력 중
- 전송 중
- 입력 오류

### UX 메모

- 첫 화면은 “무엇을 물어보면 되는지”가 바로 이해되어야 한다.
- 예시 질문 문구를 플레이스홀더 또는 보조 문구로 제공하는 것을 고려한다.

---

## 6.3 Analysis Loading Page

### 목적

review preview 생성 대기 중 이탈을 줄이고 사용자가 현재 상태를 이해하도록 돕는다.

### 주요 요소

- 분석 진행 상태 표시
- 로딩 애니메이션
- 진행 중 메시지
- 사용자 흥미를 유지할 보조 정보 영역
- review task 기준 현재 draft 정보
- 실패 시 재시도 액션
- 완료 시 결과 보기 액션

### 필수 동작

- 사용자가 현재 review preview 생성이 진행 중임을 명확히 인지할 수 있어야 한다.
- 진행 상태는 전역 review task store 기준으로 유지되어야 한다.
- active review task가 있으면 해당 draft 기준으로 복귀할 수 있어야 한다.
- preview 준비 후 결과 보기 액션을 제공해야 한다.
- 실패 시 다시 시도와 홈 복귀 액션을 제공해야 한다.

### 상태

- 분석 준비 중
- 질의 정제 중
- 기사 수집 중
- 추가 stage 메시지 표시 중
- 완료
- 실패

### UX 메모

- 단순 스피너만 두지 않는다.
- 진행 단계 메시지나 정보 카드, 팩트체크 관련 짧은 안내 문구 등을 활용해 체감 대기 시간을 줄인다.
- 현재 구현에서는 실제 pipeline 상태와 별도로 안내용 stage 메시지를 함께 노출한다.

## 6.4 Analysis Result Page

### 목적

사용자가 query-processing preview 단계에서 수집된 근거와 그 근거를 바탕으로 계산한 임시 분석 결과를 함께 확인하고 이후 판단을 위한 맥락을 이해하도록 돕는다.

### 주요 요소

- 사용자가 입력한 질문
- provisional verdict / confidence / consensus
- 수집 뉴스 종합 요약
- 공식 출처 확인 상태
- source raw snippet / evidence signal 기반 근거 목록
- source 목록
- source type filter
- 기본 숨김 query context / generated queries
- 한국 관련성 / topic scope / topic country 판정 결과
- insufficiency / pending 안내

### 필수 동작

- 현재 단계가 최종 interpretation 저장 결과가 아니라 preview artifact 기반 임시 결과임을 명시해야 한다.
- 수집된 source를 링크 형태로 제공해야 한다.
- 사용자는 source type별로 목록을 필터링할 수 있어야 한다.
- 검색 결과 스니펫, source metadata, evidence signal 기반 근거를 함께 스캔할 수 있어야 한다.
- normalized claim, topic scope, topic country, country detection reason은 기본 숨김 query context를 펼쳐 확인할 수 있어야 한다.
- interpretation과 최종 verdict가 아직 준비되지 않았고, 현재 값은 임시 파생 결과임을 안내해야 한다.

### 상태

- preview 생성 완료 / 임시 result 계산 완료
- 일부 데이터 부족
- 출처 부족
- preview 생성 실패

### UX 메모

- “AI가 이렇게 말했다”보다 “어떤 근거를 모았는지”가 더 중요하게 보여야 한다.
- 현재 구현은 최종 truth 판정이 아니라 근거 기반 임시 해석 화면이라는 점을 숨기지 않는다.
- 차트/표가 들어가더라도 source snippet과 source card 중심 구조를 우선한다.
- 모바일에서 source metadata와 snippet이 쉽게 스캔 가능해야 한다.

## 6.5 Popular Page

### 목적

최근 24시간 동안 새 검토 요청과 의미 있는 재진입이 함께 모인 주제를 랭킹 형태로 보여준다.

### 주요 요소

- 인기 질문 랭킹 리스트
- 순위 표시
- 질문 제목 또는 요약
- 요청 수, 재열람 수, 합산 점수

### 필수 동작

- 최근 24시간 합산 점수가 10 이상인 topic만 노출해야 한다.
- 합산 점수는 `submitted + meaningful reopen`으로 계산해야 한다.
- meaningful reopen은 `popular`, `history`, `notification` 재진입만 포함해야 한다.
- 각 항목은 선택 가능해야 한다.
- 정치·경제 지원 범위 안의 topic만 인기 항목으로 노출해야 한다.
- 선택 시 새 분석을 만들지 않고 기존 대표 review preview로 이동할 수 있어야 한다.

### 상태

- 데이터 있음
- 데이터 없음
- 갱신 중

### UX 메모

- 트렌드 소비 화면처럼 빠르게 훑을 수 있어야 한다.
- 순위, 주제, 관심도 정보가 한눈에 보이게 구성한다.
- 관심도 정보는 요청 수와 재열람 수를 함께 보여 주되, 점수 기준을 사용자가 이해할 수 있어야 한다.

## 6.6 Community Page

### 목적

사용자들이 주제에 대해 공개적으로 토론하고 의견을 나누는 공간이다.

### 주요 요소

- 게시글 목록
- 게시글 작성 버튼
- 작성자 정보
  - 실명
  - 성별
  - 나이대
- 댓글 영역
- 반응 기능

### 필수 동작

- 게시판 형태로 글 목록을 제공한다.
- 게시글과 댓글 작성 시 익명은 허용하지 않는다.
- 각 사용자 정보는 실명, 성별, 나이대 기준으로 노출된다.

### 상태

- 목록 조회
- 게시글 없음
- 게시글 작성
- 댓글 작성
- 신고 또는 제재 상태

### UX 메모

- 신뢰성과 책임감을 강조하는 분위기가 필요하다.
- 과도하게 가볍거나 밈 중심인 UX보다 공적 토론에 가까운 느낌이 적합하다.

## 6.7 History Page

### 목적

사용자가 이전에 수행한 질문 및 분석 기록을 확인할 수 있도록 한다.

### 주요 요소

- 과거 질문 리스트
- 질문 일시
- 분석 상태
- 결과 재진입 액션
- history 재진입은 meaningful reopen으로 기록될 수 있어야 한다.

### 필수 동작

- 사용자는 자신의 이전 분석 이력을 볼 수 있어야 한다.
- 특정 항목을 선택하면 기존 결과를 다시 볼 수 있어야 한다.

### 진입 경로

- 상단 햄버거 메뉴

### UX 메모

- 최근 항목 우선 정렬이 기본이다.
- 질문 내용이 길 경우 잘라서 보여주되 재진입 시 전체 확인이 가능해야 한다.

## 6.8 Settings Page

### 목적

사용자 관련 메뉴와 향후 설정 확장 진입점을 제공한다.

### 주요 요소

- 내 정보 관리
- 개인정보 보호
- 알림 설정
- 고객 센터
- 로그아웃 버튼

### 필수 동작

- 설정 화면에서는 사용자 정보 관리, 개인정보 보호, 알림 설정, 고객 센터 진입을 제공한다.
- 개인정보 보호는 현재 MVP 범위의 수집 정보와 사용 원칙을 안내해야 한다.
- 알림 설정은 review 완료, 커뮤니티 댓글, 커뮤니티 좋아요 알림 토글을 제공해야 한다.
- 세 알림 토글은 모두 실제 서버 알림 생성 여부에 연결되어야 한다.
- 고객 센터는 FAQ와 임시 문의 안내를 제공하고 실제 문의 접수는 제외한다.
- 추후 설정 항목이 늘어날 수 있으나 MVP에서는 정보형 페이지와 최소 설정만 유지한다.

## 6.10 Notifications Page / Panel

### 목적

사용자에게 중요한 이벤트를 즉시 전달하는 패널을 제공한다.

### 알림 종류

- review preview 완료 알림
- 커뮤니티 댓글 알림
- 커뮤니티 좋아요 알림

### 필수 동작

- 사용자는 상단 우측 알림 버튼으로 알림 목록을 확인할 수 있어야 한다.
- 알림 목록은 서버 API를 기준으로 조회해야 한다.
- review 완료, 댓글, 좋아요 알림이 설정값에 따라 실제로 생성되어야 한다.
- 알림 클릭 시 관련 review preview 화면으로 이동할 수 있어야 한다.
- 커뮤니티 알림 클릭 시 관련 게시글 상세 화면으로 이동할 수 있어야 한다.
- 읽음 / 모두 읽음 처리가 가능해야 한다.

### UX 메모

- 미확인 알림 여부가 직관적으로 보여야 한다.
- 알림은 단순 정보 전달이 아니라 바로 행동으로 이어질 수 있어야 한다.
- community / system 타입은 현재 UI 구조만 존재하며 실제 서버 연동 상태처럼 보이지 않게 표현해야 한다.

## 7. 공통 UI/UX 원칙

## 7.1 디자인 원칙

- 복잡하게 꾸미기보다 신뢰감, 가독성, 명확성을 우선한다.
- 정보 서비스다운 차분한 구조를 유지한다.
- 사용자의 질문 → 분석 → 결과 확인 흐름을 가장 짧게 만든다.

## 7.2 컬러 원칙

- 메인 컬러: 신뢰감을 주는 블루
- 보조 색상은 메인 블루를 해치지 않는 중립 계열 위주로 사용
- 경고, 실패, 성공 상태는 의미 중심으로 구분

## 7.3 반응형 원칙

- 모바일 우선 설계
- 데스크톱에서는 콘텐츠 폭을 적절히 제한
- 차트, 표, 링크 리스트는 작은 화면에서도 읽기 쉬워야 한다

## 7.4 다크모드 정책

- 초기 버전에서는 다크모드를 지원하지 않는다.
