# VARO Product Spec

## 1. 문서 개요

### 1.1 목적
이 문서는 VARO 서비스의 화면 구조, 사용자 흐름, 주요 페이지 요구사항, UI/UX 방향을 정의한다.  
VARO는 사용자가 뉴스, 주장, 이슈에 대해 질문하면 관련 기사 및 소셜 출처를 수집하고, 이를 시각화하여 사용자가 객관적으로 판단할 수 있도록 돕는 모바일 중심 반응형 웹 서비스다.

### 1.2 범위
본 문서는 다음 범위를 포함한다.

- Information Architecture
- Navigation Structure
- User Flow
- Page Specification
- UI/UX 방향
- 기본 정책

### 1.3 제품 방향
- 모바일 웹 우선 설계
- 반응형 웹 지원
- 신뢰감 있는 정보 탐색 경험 제공
- 복잡한 기능보다 빠른 질문, 분석, 확인 흐름에 집중
- 초기 버전에서는 다크모드 미지원

## 2. 기본 정책

### 2.1 플랫폼
- 서비스 형태: 모바일 웹
- 지원 방식: 반응형 웹
- 우선 기준: 모바일 퍼스트
- 데스크톱에서도 사용 가능해야 하나, 핵심 UX는 모바일 기준으로 설계한다.

### 2.2 인증 정책
- 로그인은 필수다.
- 홈 진입 시 비로그인 상태라면 로그인 페이지로 리다이렉트한다.
- 로그인 방식은 **구글 로그인만 지원**한다.
- 초기 버전에서는 이메일 로그인, 자체 회원가입, SNS 추가 로그인은 지원하지 않는다.

### 2.3 테마 정책
- 대표 컬러는 **신뢰감을 주는 중간 톤의 블루**를 사용한다.
- 너무 어둡거나 너무 밝은 블루는 지양한다.
- 다크모드는 지원하지 않는다.
- 전체 UI는 명확하고 밝은 기본 테마를 유지한다.

### 2.4 커뮤니티 정책
- 커뮤니티에서는 익명 사용을 허용하지 않는다.
- 모든 사용자는 **실명, 성별, 나이대**를 공개한 상태로 활동해야 한다.
- 커뮤니티의 신뢰성과 책임성을 강화하는 방향으로 운영한다.

### 2.5 사용자 정보 정책
- 사용자 정보 화면에서 다음 정보를 확인할 수 있다.
  - 실명
  - 성별
  - 나이
  - 사는 나라
  - 사는 도시
- 이 중 **사는 나라, 사는 도시만 수정 가능**하다.
- 실명, 나이, 성별은 수정할 수 없다.

## 3. Information Architecture

## 3.1 전체 화면 구조
- Login
- Home
- Analysis Loading
- Analysis Result
- Popular
- Community
- History
- User Settings / User Info
- Notifications

## 3.2 주요 페이지 설명
| 페이지 | 목적 |
|---|---|
| Login | 구글 로그인 수행 |
| Home | 사용자가 질문을 입력하고 분석을 시작 |
| Analysis Loading | 분석 진행 중 상태 제공 |
| Analysis Result | 출처, 시각화, 분석 결과 제공 |
| Popular | 많이 질문된 주제와 순위 제공 |
| Community | 사용자 간 게시글/댓글 기반 토론 |
| History | 사용자의 이전 질문 및 분석 기록 확인 |
| User Info | 사용자 정보 확인 및 일부 수정 |
| Notifications | 분석 완료, 커뮤니티 알림 확인 |

## 4. Navigation Structure

## 4.1 하단 네비게이션
하단 네비게이션에는 다음 3개만 노출한다.

- Home
- 인기
- 커뮤니티

## 4.2 상단 버튼 구조
상단에는 다음 버튼을 둔다.

- 좌측: 햄버거 메뉴 버튼
- 우측: 알림 버튼

## 4.3 햄버거 메뉴
햄버거 메뉴를 열면 다음 항목을 확인할 수 있다.

- 히스토리
- 설정

설정 화면 내부에는 사용자 정보 확인 버튼이 있다.

## 4.4 알림 버튼
알림 버튼은 다음 용도로 사용한다.

- 커뮤니티 관련 알림
- 사용자가 요청한 분석 완료 알림

## 5. User Flows

## 5.1 로그인 진입 흐름
1. 사용자가 서비스에 접속한다.
2. 시스템은 로그인 여부를 확인한다.
3. 비로그인 상태라면 로그인 페이지로 이동시킨다.
4. 사용자는 구글 로그인 버튼을 눌러 인증한다.
5. 로그인 성공 시 Home으로 이동한다.

## 5.2 질문 분석 흐름
1. 사용자가 Home 화면 중앙 채팅박스에 질문을 입력한다.
2. 사용자가 질문 전송 액션을 수행한다.
3. 시스템은 분석 요청을 생성한다.
4. 사용자는 Analysis Loading 화면으로 이동한다.
5. 시스템은 기사/소셜 출처를 수집하고 분석한다.
6. 분석 완료 후 Analysis Result 화면으로 이동한다.
7. 사용자는 출처 링크, 차트, 표, 분석 결과를 확인한다.

## 5.3 분석 완료 알림 흐름
1. 사용자가 질문을 입력하고 분석을 요청한다.
2. 분석이 일정 시간 이상 소요된다.
3. 시스템은 분석 완료 시 알림을 생성한다.
4. 사용자는 알림 버튼을 통해 완료 여부를 확인할 수 있다.
5. 사용자는 알림을 눌러 결과 화면으로 이동할 수 있다.

## 5.4 인기 화면 탐색 흐름
1. 사용자가 하단 네비게이션에서 인기 탭을 선택한다.
2. 시스템은 현재 가장 많이 질문된 주제를 순위 형태로 보여준다.
3. 사용자는 인기 항목을 눌러 해당 주제 상세 또는 분석 화면으로 이동할 수 있다.

## 5.5 커뮤니티 이용 흐름
1. 사용자가 하단 네비게이션에서 커뮤니티 탭을 선택한다.
2. 게시글 목록을 확인한다.
3. 게시글 작성자 정보로 실명, 성별, 나이대가 노출된다.
4. 사용자는 게시글 상세 진입, 댓글 작성, 반응 확인 등을 수행한다.
5. 관련 활동이 발생하면 알림으로 전달될 수 있다.

## 5.6 히스토리 및 사용자 정보 확인 흐름
1. 사용자가 상단 좌측 햄버거 메뉴를 연다.
2. 히스토리 또는 설정을 선택한다.
3. 설정 화면에서 사용자 정보 버튼을 누른다.
4. 사용자 정보 화면에서 프로필 정보를 확인한다.
5. 사는 나라, 사는 도시만 수정 가능하다.

## 6. Page Specifications

## 6.1 Login Page

### 목적
비로그인 사용자를 인증시키고 서비스 진입을 제어한다.

### 주요 요소
- 서비스 로고
- 서비스 한 줄 소개
- 구글 로그인 버튼

### 필수 동작
- 로그인하지 않은 사용자는 Home에 직접 접근할 수 없다.
- 로그인 페이지에서 구글 로그인만 제공한다.
- 로그인 성공 시 Home으로 이동한다.

### 상태
- 기본 상태
- 로그인 진행 중
- 로그인 실패
- 로그인 성공

### UX 메모
- 첫 진입에서 복잡한 설명보다 서비스 신뢰감과 간결함을 우선한다.
- CTA는 구글 로그인 버튼 하나만 강조한다.

---

## 6.2 Home Page

### 목적
사용자가 질문을 입력하고 분석을 시작하는 핵심 진입 화면이다.

### 주요 요소
- 상단 바
  - 좌측 햄버거 메뉴 버튼
  - 우측 알림 버튼
- 중앙 채팅박스
- 질문 입력 영역
- 질문 전송 액션

### 핵심 구조
- 화면 정중앙에는 채팅박스만 강조한다.
- 사용자가 자연스럽게 질문을 입력하도록 단순한 구조를 유지한다.
- 초기 화면에는 불필요한 정보나 과한 카드 노출을 지양한다.

### 필수 동작
- 질문 입력 가능
- 질문 전송 시 분석 요청 생성
- 요청 후 Analysis Loading 화면으로 이동

### 상태
- 입력 전
- 입력 중
- 전송 중
- 입력 오류

### UX 메모
- 첫 화면은 “무엇을 물어보면 되는지”가 바로 이해되어야 한다.
- 예시 질문 문구를 플레이스홀더 또는 보조 문구로 제공하는 것을 고려한다.

---

## 6.3 Analysis Loading Page

### 목적
분석 대기 중 이탈을 줄이고 사용자가 지루하지 않게 기다리도록 돕는다.

### 주요 요소
- 분석 진행 상태 표시
- 로딩 애니메이션
- 진행 중 메시지
- 사용자 흥미를 유지할 보조 정보 영역

### 필수 동작
- 사용자가 현재 분석이 진행 중임을 명확히 인지할 수 있어야 한다.
- 결과가 준비되면 자동 또는 액션 기반으로 결과 화면으로 전환한다.
- 분석이 오래 걸리는 경우 알림으로 완료 사실을 알려줄 수 있어야 한다.

### 상태
- 분석 준비 중
- 기사 수집 중
- 소셜 데이터 정리 중
- 시각화 생성 중
- 완료
- 실패

### UX 메모
- 단순 스피너만 두지 않는다.
- 진행 단계 메시지나 정보 카드, 팩트체크 관련 짧은 안내 문구 등을 활용해 체감 대기 시간을 줄인다.

---

## 6.4 Analysis Result Page

### 목적
사용자가 분석 결과와 근거 자료를 직접 확인하고 객관적으로 판단할 수 있도록 한다.

### 주요 요소
- 사용자가 입력한 질문
- 분석 결과 요약
- 기사 출처 링크 목록
- 소셜 출처 링크 목록
- 차트
- 표
- 핵심 인사이트 영역

### 필수 동작
- 수집된 기사 및 소셜 출처를 링크 형태로 제공해야 한다.
- 출처별로 사용자가 원문을 확인할 수 있어야 한다.
- 정보는 차트나 표 등 시각적 방식으로 정리되어야 한다.
- 결과는 텍스트 요약만이 아니라 구조화된 정보로 제시해야 한다.

### 상태
- 결과 생성 완료
- 일부 데이터 부족
- 출처 부족
- 분석 실패

### UX 메모
- “AI가 이렇게 말했다”보다 “어떤 근거를 모았는지”가 더 중요하게 보여야 한다.
- 출처 리스트, 요약 카드, 시각화 영역이 위계 있게 배치되어야 한다.
- 모바일에서도 표와 차트가 무너지지 않도록 반응형 설계를 해야 한다.

---

## 6.5 Popular Page

### 목적
현재 사용자들이 가장 많이 질문하는 주제를 랭킹 형태로 보여준다.

### 주요 요소
- 인기 질문 랭킹 리스트
- 순위 표시
- 질문 제목 또는 요약
- 관련 카운트 정보

### 필수 동작
- 사람들이 현재 가장 자주 질문하는 주제를 순위로 제공한다.
- 각 항목은 선택 가능해야 한다.
- 선택 시 해당 주제 관련 상세 화면 또는 분석 결과로 이동할 수 있어야 한다.

### 상태
- 데이터 있음
- 데이터 없음
- 갱신 중

### UX 메모
- 트렌드 소비 화면처럼 빠르게 훑을 수 있어야 한다.
- 순위, 주제, 관심도 정보가 한눈에 보이게 구성한다.

---

## 6.6 Community Page

### 목적
사용자들이 주제에 대해 공개적으로 토론하고 의견을 나누는 공간이다.

### 주요 요소
- 게시글 목록
- 게시글 작성 버튼
- 작성자 정보
  - 실명
  - 성별
  - 나이대
- 댓글 영역
- 반응 기능

### 필수 동작
- 게시판 형태로 글 목록을 제공한다.
- 게시글과 댓글 작성 시 익명은 허용하지 않는다.
- 각 사용자 정보는 실명, 성별, 나이대 기준으로 노출된다.

### 상태
- 목록 조회
- 게시글 없음
- 게시글 작성
- 댓글 작성
- 신고 또는 제재 상태

### UX 메모
- 신뢰성과 책임감을 강조하는 분위기가 필요하다.
- 과도하게 가볍거나 밈 중심인 UX보다 공적 토론에 가까운 느낌이 적합하다.

---

## 6.7 History Page

### 목적
사용자가 이전에 수행한 질문 및 분석 기록을 확인할 수 있도록 한다.

### 주요 요소
- 과거 질문 리스트
- 질문 일시
- 분석 상태
- 결과 재진입 액션

### 필수 동작
- 사용자는 자신의 이전 분석 이력을 볼 수 있어야 한다.
- 특정 항목을 선택하면 기존 결과를 다시 볼 수 있어야 한다.

### 진입 경로
- 상단 햄버거 메뉴

### UX 메모
- 최근 항목 우선 정렬이 기본이다.
- 질문 내용이 길 경우 잘라서 보여주되 재진입 시 전체 확인이 가능해야 한다.

---

## 6.8 Settings Page

### 목적
사용자 관련 설정 진입점을 제공한다.

### 주요 요소
- 사용자 정보 버튼

### 필수 동작
- 설정 화면에서는 우선 사용자 정보 확인 기능만 제공한다.
- 추후 설정 항목이 늘어날 수 있으나 MVP에서는 단순하게 유지한다.

---

## 6.9 User Info Page

### 목적
사용자의 프로필 정보를 조회하고 일부 항목만 수정할 수 있도록 한다.

### 주요 요소
- 실명
- 성별
- 나이
- 사는 나라
- 사는 도시
- 수정 버튼

### 수정 가능 항목
- 사는 나라
- 사는 도시

### 수정 불가 항목
- 실명
- 성별
- 나이

### UX 메모
- 수정 가능한 항목과 불가능한 항목이 명확히 구분되어야 한다.
- 변경 불가 항목은 비활성 또는 읽기 전용 형태로 표시한다.

## 6.10 Notifications Page / Panel

### 목적
사용자에게 중요한 이벤트를 즉시 전달한다.

### 알림 종류
- 분석 완료 알림
- 커뮤니티 관련 알림

### 필수 동작
- 사용자는 상단 우측 알림 버튼으로 알림 목록을 확인할 수 있어야 한다.
- 알림 클릭 시 관련 화면으로 이동할 수 있어야 한다.

### UX 메모
- 미확인 알림 여부가 직관적으로 보여야 한다.
- 알림은 단순 정보 전달이 아니라 바로 행동으로 이어질 수 있어야 한다.

## 7. 공통 UI/UX 원칙

## 7.1 디자인 원칙
- 복잡하게 꾸미기보다 신뢰감, 가독성, 명확성을 우선한다.
- 정보 서비스다운 차분한 구조를 유지한다.
- 사용자의 질문 → 분석 → 결과 확인 흐름을 가장 짧게 만든다.

## 7.2 컬러 원칙
- 메인 컬러: 신뢰감을 주는 블루
- 보조 색상은 메인 블루를 해치지 않는 중립 계열 위주로 사용
- 경고, 실패, 성공 상태는 의미 중심으로 구분

## 7.3 반응형 원칙
- 모바일 우선 설계
- 데스크톱에서는 콘텐츠 폭을 적절히 제한
- 차트, 표, 링크 리스트는 작은 화면에서도 읽기 쉬워야 한다

## 7.4 다크모드 정책
- 초기 버전에서는 다크모드를 지원하지 않는다.
